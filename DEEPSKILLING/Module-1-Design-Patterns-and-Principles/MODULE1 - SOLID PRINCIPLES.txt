SOLID PRINCIPLES 

S -> Single Responsibility Principle (SRP)

This principle states that a class should have only one responsibility or one reason to change. If a class performs multiple tasks, any change in one task may affect the others, making the code difficult to maintain.

Example:
Consider a WashingMachine. Its job is to wash clothes. It should not also be responsible for billing customers or sending notifications. If washing logic changes, only the WashingMachine class should be modified.

Real-life example:

 Washing Machine → Washes clothes
 Billing System → Generates bills
 Notification System → Sends messages

Each component has a single responsibility.

O -> Open/Closed Principle (OCP)

This principle states that software should be open for extension but closed for modification. New features should be added without changing existing code.

Example:
Consider an Oven.

Initially, it supports:

 Bake Mode

Later, the company wants:

 Grill Mode
 Pizza Mode

Instead of modifying the existing Bake functionality new modes can be added separately.

Real-life example:
A smartphone gets new apps installed without changing the phone's operating system. The phone is extended with new functionality rather than modifying the existing features.


L -> Liskov Substitution Principle (LSP)

This principle states that a child class should be able to replace its parent class without causing incorrect behavior.

Example:
Suppose we have a Woman class.

A Mother is a Woman.
A Teacher is a Woman.

Anywhere a Woman is expected, a Mother or Teacher can be used because they still behave as a Woman.

Bad Example:
Suppose a Bird class has a fly() method.
A Sparrow can fly, so it can replace Bird.
A Penguin cannot fly, so if Penguin extends Bird and fly() fails, LSP is violated.

Similarly, if a Vehicle class assumes every vehicle uses petrol ,an ElectricCar cannot properly replace it because electric cars do not use petrol.


I -> Interface Segregation Principle (ISP)

This principle states that a class should not be forced to implement methods that it does not need. Large interfaces should be split into smaller and specific interfaces.

Example:
Consider household appliances.

Interface:

 Wash()
 Bake()
 Cool()

A Washing Machine only washes.
An Oven only bakes.
A Refrigerator only cools.

Forcing all appliances to implement all three methods makes no sense.

Instead:

Washable → Wash()
Bakable → Bake()
Coolable → Cool()

Now each appliance implements only the functionality it needs.


D -> Dependency Inversion Principle (DIP)

This principle states that high-level modules should not depend directly on low-level modules. Both should depend on abstractions.

Example:
Consider a Woman ordering food using a Food Delivery App.

Bad Design:
The app depends directly on a specific Restaurant.

If the restaurant closes, the app must be modified.

Good Design:
The app depends on a FoodProvider interface.

Different restaurants implement FoodProvider:

 Restaurant A
 Restaurant B
 Restaurant C

The app can work with any restaurant without changing its code.

Another Example:
A Notification system should depend on a MessageService interface rather than directly on EmailService. Then we can easily switch to SMS or WhatsApp services without changing the Notification class.
